Sistem Pakar Diagnosis
Kerusakan Laptop/PC

Aplikasi web berbasis Forward Chaining + Certainty Factor untuk mendiagnosis kerusakan hardware laptop dan PC secara interaktif.

🗓️ Versi 1.0.0 🐍 Python 3.11+ / FastAPI ⚛️ Next.js 14+ / TypeScript 🗄️ SQLite / PostgreSQL
DiagLaptop / › Dokumentasi Teknis

Gambaran Umum

DiagLaptop adalah sistem pakar berbasis web yang membantu pengguna mendiagnosis kerusakan hardware laptop dan PC. Pengguna menjawab serangkaian pertanyaan tentang gejala yang dialami perangkatnya, lalu sistem menggunakan inference engine Forward Chaining dikombinasikan dengan Certainty Factor (CF) untuk menyimpulkan kemungkinan kerusakan beserta langkah solusinya.

Tujuan Utama
  • Mendiagnosis kerusakan hardware secara otomatis
  • Memberikan rekomendasi solusi yang konkret
  • Mengukur tingkat kepercayaan (CF) tiap diagnosis
  • Menyimpan riwayat diagnosis untuk analisis
Fitur Utama
  • Wizard diagnosis per kategori gejala
  • 9+ kategori kerusakan hardware
  • Panel admin CRUD (gejala, kerusakan, rules)
  • Riwayat diagnosis dengan session ID
  • Autentikasi JWT untuk admin

Alur Sistem Secara Keseluruhan

1
User membuka halaman Diagnosis

Frontend fetch daftar gejala dari GET /api/v1/gejala

2
User menjawab setiap gejala

Hasil ditampilkan

Maks. 5 kerusakan teratas beserta deskripsi, solusi, dan persentase CF

Inference Engine

Inti sistem pakar ini terdiri dari dua modul di backend/app/engine/: perhitungan Certainty Factor dan Forward Chaining.

Certainty Factor (CF)

CF mengukur tingkat kepercayaan suatu diagnosis. Nilai berkisar dari −1.0 (pasti tidak) hingga 1.0 (pasti ya).

Formula CF

# Langkah 1: Gabungkan CF pakar dan CF user
CF_combined = CF_pakar × CF_user

# Langkah 2: Merge CF dari banyak gejala untuk satu kerusakan
# Keduanya positif:
CF_merge = CF1 + CF2 × (1 − CF1)

# Keduanya negatif:
CF_merge = CF1 + CF2 × (1 + CF1)

# Berlawanan tanda:
CF_merge = (CF1 + CF2) / (1 − min(|CF1|, |CF2|))

Implementasi Python

# certainty_factor.py

def cf_combine(cf_pakar: float, cf_user: float) -> float:
    """Langkah 1: CF pakar × CF user"""
    return cf_pakar * cf_user


def cf_merge(cf1: float, cf2: float) -> float:
    """Langkah 2: Gabungkan dua nilai CF dengan formula standar"""
    if cf1 >= 0 and cf2 >= 0:
        return cf1 + cf2 * (1 - cf1)
    elif cf1 < 0 and cf2 < 0:
        return cf1 + cf2 * (1 + cf1)
    else:
        denom = 1 - min(abs(cf1), abs(cf2))
        if denom == 0.0: return 0.0
        return (cf1 + cf2) / denom


def merge_cf_list(cf_values: list[float]) -> float:
    """Merge daftar CF secara sekuensial (kiri ke kanan)"""
    if not cf_values: return 0.0
    result = cf_values[0]
    for cf in cf_values[1:]:
        result = cf_merge(result, cf)
    return result

Forward Chaining

Proses inferensi di forward_chain.py:

def run_diagnosis(db: Session, gejala_inputs: list[GejalaInput]) -> list[DiagnosisItem]:
    # 1. Ambil semua rule yang cocok dengan gejala yang dipilih
    gejala_ids = [g.gejala_id for g in gejala_inputs]
    rules = db.query(Rule).filter(Rule.gejala_id.in_(gejala_ids)).all()

    # 2. Hitung CF combined per rule, kumpulkan per kerusakan
    kerusakan_cf_map: dict[str, list[float]] = {}
    for rule in rules:
        cf_user = cf_user_map.get(rule.gejala_id, 0.0)
        cf_combined = cf_combine(rule.cf_pakar, cf_user)
        if cf_combined == 0.0: continue
        kerusakan_cf_map.setdefault(rule.kerusakan_id, []).append(cf_combined)

    # 3. Merge semua CF per kerusakan, filter CF < 0.2
    for kerusakan_id, cf_list in kerusakan_cf_map.items():
        cf_total = merge_cf_list(cf_list)
        if cf_total < CF_THRESHOLD: continue  # threshold = 0.2

    # 4. Sort descending, return top 5
    results.sort(key=lambda x: x.cf_total, reverse=True)
    return results[:MAX_RESULTS]  # MAX_RESULTS = 5
⚠️
Konstanta penting: CF_THRESHOLD = 0.2 dan MAX_RESULTS = 5 didefinisikan di forward_chain.py. Gejala dengan cf_user = 0.0 (Tidak Tahu) tidak berkontribusi ke kalkulasi apapun karena CF_combined = cf_pakar × 0 = 0.

Contoh Kalkulasi

GejalaCF PakarJawaban UserCF UserCF Combined
G001 — Tidak menyala0.9Ya1.00.9
G002 — LED tidak nyala0.8Mungkin0.60.48
G007 — Adaptor mati0.95Ya1.00.95

Misalnya ketiga gejala di atas sama-sama menunjuk ke kerusakan Adaptor Rusak:

CF_merge(0.9, 0.48) = 0.9 + 0.48 × (1 − 0.9) = 0.9 + 0.048 = 0.948
CF_merge(0.948, 0.95) = 0.948 + 0.95 × (1 − 0.948) = 0.948 + 0.0494 ≈ 0.9974

const api = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_URL || "http://localhost:8000",
  headers: { "Content-Type": "application/json" },
});

// Request interceptor — inject Bearer token
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("token");
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Response interceptor — auto logout on 401
api.interceptors.response.use(
  (res) => res,
  (err) => {
    if (err.response?.status === 401) {
      localStorage.removeItem("token");
      window.location.href = "/login";
    }
    return Promise.reject(err);
  }
);

Konvensi Kode

TypeScript / Next.js

  • Gunakan App Router (bukan Pages Router)
  • Semua komponen UI pakai shadcn/ui terlebih dahulu
  • API calls hanya di lib/api.ts, tidak inline di komponen
  • Tipe data shared di types/index.ts
  • Gunakan "use client" hanya bila perlu (hooks, interaktivitas)
  • Nama file komponen: PascalCase (QuestionWizard.tsx)
  • Nama file utils/lib: camelCase (api.ts, store.ts)

Python / FastAPI

  • Semua endpoint menggunakan async/await
  • Schema Pydantic terpisah dari ORM model
  • Dependency injection via Depends() untuk DB session dan auth
  • Response selalu menggunakan schema Pydantic, bukan raw dict
  • Error menggunakan HTTPException dengan status code tepat
  • Nama file: snake_case (forward_chain.py)
  • Linter: Ruff + Black

Alur Penambahan Fitur Baru

  1. Tambah model SQLAlchemy di backend/app/models/
  2. Tambah Pydantic schema di backend/app/schemas/
  3. Tambah endpoint di backend/app/api/v1/ dan daftarkan ke router.py
  4. Tambah TypeScript interface di frontend/types/index.ts
  5. Tambah fungsi API di frontend/lib/api.ts
  6. Buat halaman/komponen di frontend/app/ atau frontend/components/

Glosarium Domain

IstilahMakna
gejalaSymptom hardware yang dialami user — ditanyakan satu per satu selama proses diagnosis
kerusakanJenis kerusakan hardware yang didiagnosis oleh sistem (mis. "Adaptor Rusak", "RAM Bermasalah")
ruleRelasi antara satu gejala dengan satu kerusakan, disertai nilai kepercayaan pakar (cf_pakar)
cf_pakarCertainty Factor dari pakar — nilai antara 0.0 dan 1.0 yang merepresentasikan seberapa kuat gejala menunjuk ke kerusakan tersebut
cf_userCertainty Factor dari jawaban user — Ya=1.0, Mungkin=0.6, Tidak Tahu=0.0, Tidak=−1.0
cf_combinedHasil perkalian cf_pakar × cf_user — kontribusi satu gejala terhadap satu kerusakan
cf_totalCF akhir setelah semua gejala digabungkan via merge_cf_list() untuk satu kerusakan tertentu
diagnosisProses inferensi lengkap: dari input gejala → running CF → output hasil terurut
solusiLangkah perbaikan yang disarankan untuk setiap kerusakan yang terdeteksi
session_idUUID unik per sesi diagnosis — digunakan untuk tracking riwayat
knowledge baseKumpulan gejala, kerusakan, dan rules yang merepresentasikan pengetahuan pakar hardware
forward chainingStrategi inferensi berbasis data — dimulai dari fakta (gejala yang dipilih) menuju kesimpulan (kerusakan)
